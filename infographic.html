Logic: Combine all pending changes into one big test batch to save resources.

The Problem: Since Change C is bad, the entire batch fails.

Consequence: The system doesn't know who caused it. It must reject everyone (A, B, D, E are innocent victims) or start a slow "bisect" process to find C.

Throughput drops to zero whenever a bug is submitted.

⬇ Distribute into Overlapping Minibatches ⬇

Logic: Assign each change to multiple random small batches (e.g., 3 batches each).

The Magic: Change C fails every batch it touches. Innocent changes (like A) might share a failing batch with C, but they also exist in passing batches elsewhere.

Consequence: We calculate a "Suspicion Score". C has 100% failure rate. A has 33% (unlucky). The system correctly identifies and rejects only C.
